# Requesting Code Review - 代码审查

## 何时使用

当满足以下条件时，**必须**使用此技能：

1. **任务完成** - 完成一个任务后，切换到下一个任务前
2. **功能实现** - 实现了某个功能后
3. **修复完成** - 修复了某个bug后

## 代码审查原则

### 1. 及时审查

不要积累太多代码才审查：
- 每个任务完成后立即审查
- 少量代码更容易发现问题

### 2. 关注重点

审查时关注最重要的方面：
- 功能正确性
- 代码安全性
- 性能问题

### 3. 按严重程度分类

将问题按严重程度分类：

```
关键问题 (Critical) - 必须修复
- 安全漏洞
- 数据丢失风险
- 系统崩溃

重要问题 (Major) - 应该修复
- 逻辑错误
- 严重的性能问题
- 代码难以理解

次要问题 (Minor) - 建议修复
- 代码风格
- 小的优化
- 注释不足
```

## 审查检查清单

### 功能正确性

- [ ] 代码实现了预期功能？
- [ ] 边界情况处理正确？
- [ ] 错误处理完善？

### 代码质量

- [ ] 代码清晰易读？
- [ ] 命名有意义？
- [ ] 没有重复代码？

### 安全性

- [ ] 没有安全漏洞？
- [ ] 用户输入已验证？
- [ ] 敏感数据已保护？

### 性能

- [ ] 没有明显的性能问题？
- [ ] 数据库查询已优化？
- [ ] 大数据量已处理？

### 测试

- [ ] 有测试覆盖关键逻辑？
- [ ] 测试用例合理？

## 工作流程

### 1. 收集变更

查看本次变更的内容：
- 修改的文件
- 新增的代码
- 删除的代码

### 2. 执行审查

逐个文件、逐个函数审查：
- 理解代码逻辑
- 识别潜在问题
- 记录发现的问题

### 3. 分类问题

将发现的问题分类：

```
关键问题：
- 安全漏洞
- 致命逻辑错误

重要问题：
- 严重的性能问题
- 代码可读性差

次要问题：
- 代码风格
- 小的优化
```

### 4. 报告结果

向用户报告审查结果：

```markdown
## 代码审查报告

### 关键问题 (0)
无

### 重要问题 (2)
1. [问题描述] - 文件:src/utils.ts, 行号:45
2. [问题描述] - 文件:src/components/A.tsx, 行号:23

### 次要问题 (3)
1. [建议] - 文件:src/styles.css
2. [建议] - 文件:src/api.ts
3. [建议] - 文件:src/app.ts
```

### 5. 决定行动

根据审查结果决定：

- **有关键问题**：返回修复，阻止继续
- **有重要问题**：返回修复，建议修复后继续
- **只有次要问题**：可以继续，记录问题

## 报告模板

```markdown
## 代码审查报告

### 变更摘要
- 修改文件: X 个
- 新增代码: Y 行
- 删除代码: Z 行

### 关键问题 (X)
- [ ] [问题描述] - 文件:xxx, 行号:xxx

### 重要问题 (X)
- [ ] [问题描述] - 文件:xxx, 行号:xxx

### 次要问题 (X)
- [ ] [建议] - 文件:xxx

### 总结
- [ ] 可以合并
- [ ] 需要修复后合并
- [ ] 必须修复后合并
```

## 最佳实践

### DO

✅ **及时审查** - 完成任务后立即审查
✅ **关注重点** - 先看关键问题
✅ **分类清晰** - 按严重程度分类
✅ **提供建议** - 指出问题同时提供解决方案

### DON'T

❌ **不要忽视关键问题** - 安全和稳定性优先
❌ **不要过度批评** - 以建设性态度提出问题
❌ **不要漏掉测试** - 确保有测试覆盖
❌ **不要拖延** - 及时审查，及时反馈

## 触发条件

此技能在以下情况自动触发：

1. 完成一个任务后
2. 实现了一个功能后
3. 修复了一个bug后
4. 用户请求代码审查

## 与其他技能的配合

- **test-driven-development** - 测试覆盖关键逻辑
- **systematic-debugging** - 调试审查中发现的问题
- **verification-before-completion** - 最终验证
- **finishing-a-development-branch** - 分支完成时全面审查
