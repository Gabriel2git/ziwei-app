# Systematic Debugging - 系统调试

## 何时使用

当满足以下条件时，**必须**使用此技能：

1. **出现bug** - 代码运行结果与预期不符
2. **错误发生** - 控制台有错误信息
3. **功能异常** - 某个功能不工作

## 调试原则

### 1. 不要猜测

❌ **不要**：
- 随意修改代码期望"碰巧"修复
- 假设你知道问题在哪里
- 跳过复现步骤直接改代码

✅ **应该**：
- 先复现问题
- 收集证据
- 形成假设后再验证

### 2. 系统化方法

遵循4阶段调试流程：
1. **观察** - 收集问题信息
2. **假设** - 形成可能的原因
3. **验证** - 验证假设是否正确
4. **修复** - 修复并验证

## 4阶段调试流程

### 阶段1：观察问题

收集所有相关信息：

```
1. 错误信息
   - 错误类型和消息
   - 错误发生的位置（文件和行号）
   - 调用栈

2. 环境信息
   - 浏览器/操作系统
   - 数据输入
   - 操作步骤

3. 上下文
   - 之前是否正常工作？
   - 最近有什么改动？
   - 其他页面是否正常？
```

### 阶段2：形成假设

基于观察结果，形成可能的假设：

```
可能的原因1: API返回数据格式变化
可能的原因2: 状态管理问题
可能的原因3: 权限/认证问题
```

### 阶段3：验证假设

验证哪个假设是正确的：

```
验证方法：
1. 添加日志输出
2. 检查API响应
3. 使用断点调试
4. 检查数据流
```

### 阶段4：修复并验证

修复问题并确认不再出现：

```
1. 应用修复
2. 验证修复有效
3. 确保没有引入新问题
4. 记录问题和解决方案
```

## 常用调试技术

### 1. 日志输出

在关键位置添加日志：

```typescript
console.log('API Response:', data);
console.log('State:', this.state);
```

### 2. 断点调试

在浏览器中设置断点：
- Chrome DevTools
- VS Code 调试器

### 3. API 检查

使用网络面板检查：
- 请求是否正确发送
- 响应状态码
- 响应数据格式

### 4. 状态检查

检查应用状态：
- React DevTools
- Redux/Vue DevTools

## 调试检查清单

每次调试时检查：

- [ ] 错误信息是什么？
- [ ] 错误在哪里发生？
- [ ] 什么操作导致错误？
- [ ] 数据输入是什么？
- [ ] 之前是否正常工作？
- [ ] 最近有什么改动？

## 最佳实践

### DO

✅ **复现问题** - 首先确保能稳定复现
✅ **收集证据** - 记录错误信息和上下文
✅ **形成假设** - 基于证据提出可能原因
✅ **验证假设** - 用证据验证哪个假设正确
✅ **修复后验证** - 确保问题真正解决

### DON'T

❌ **不要猜测** - 不要随意修改代码
❌ **不要跳过步骤** - 按照调试流程来
❌ **不要只修表面** - 要找到根本原因
❌ **不要忘记验证** - 修复后要确认有效

## 常见问题模式

### 1. API 相关

```
问题：API 返回错误
检查：
- 请求URL是否正确
- 请求方法是否正确
- 请求头是否正确
- 认证/权限是否有效
```

### 2. 状态相关

```
问题：UI不更新
检查：
- 状态是否正确更新
- 组件是否正确重新渲染
- 是否有竞态条件
```

### 3. 渲染相关

```
问题：组件显示异常
检查：
- Props是否正确传递
- 样式是否正确应用
- 条件渲染是否正确
```

## 触发条件

此技能在以下情况自动触发：

1. 用户报告bug
2. 出现错误信息
3. 功能不工作
4. 测试失败

## 与其他技能的配合

- **verification-before-completion** - 修复后验证
- **test-driven-development** - 添加测试防止回归
- **requesting-code-review** - 代码审查检查潜在问题
