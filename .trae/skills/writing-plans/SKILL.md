# Writing Plans - 编写计划

## 何时使用

当满足以下条件时，**必须**使用此技能：

1. **设计已批准** - brainstorming 技能已完成设计并获得用户确认
2. **需要实施** - 用户说"开始实施"或"go"
3. **复杂任务** - 任务需要多个文件和步骤

## 典型场景

- 用户确认了设计方案，要求开始开发
- 需要将大任务拆分为可管理的小步骤
- 需要确保每个步骤都有明确的验证方法

## 工作流程

### 1. 分析设计文档

理解已批准的设计：
- 核心功能是什么？
- 需要修改哪些文件？
- 有哪些依赖关系？

### 2. 拆分为小任务

将工作分解为 2-5 分钟可完成的任务：

```
任务原则：
- 每个任务专注一件事
- 每个任务有明确的文件路径
- 每个任务有验证步骤
```

### 3. 编写任务描述

每个任务包含：
- **任务名称**：简短描述
- **具体修改**：要修改的文件和内容
- **验证步骤**：如何确认任务完成

### 4. 呈现计划

将计划分块展示给用户：
- 任务数量和预计时间
- 任务之间的依赖关系
- 关键风险点

## 任务格式示例

```markdown
## 任务 1: 创建用户认证上下文
- 文件: src/contexts/AuthContext.tsx
- 修改: 添加 AuthProvider 组件和 useAuth hook
- 验证: 编译无错误

## 任务 2: 添加登录页面
- 文件: src/app/login/page.tsx
- 修改: 创建登录表单组件
- 验证: 页面可访问，表单可提交
```

## 最佳实践

### DO

✅ **小任务原则** - 每个任务 2-5 分钟可完成
✅ **精确文件路径** - 明确告诉用户要修改哪个文件
✅ **验证步骤** - 每个任务有明确的验证方法
✅ **依赖关系** - 标注任务之间的依赖

### DON'T

❌ **不要大任务** - 超过10分钟的任务应该拆分
❌ **不要模糊路径** - 必须有具体的文件路径
❌ **不要跳过验证** - 必须有验证步骤

## 使用方法

当设计获得批准后：

1. **创建计划文件** - 在 `.trae/documents/` 目录下创建计划文件
2. **拆分任务** - 将设计拆分为小任务
3. **展示给用户** - 呈现计划，征得同意
4. **开始执行** - 用户确认后开始实施

## 计划模板

```markdown
# [任务名称] 实施计划

## 概述
[简短描述这个计划要实现什么]

## 任务列表

### 任务 1: [任务名称]
- **文件**: [精确路径]
- **修改**: [具体修改内容]
- **验证**: [验证步骤]

### 任务 2: [任务名称]
- **文件**: [精确路径]
- **修改**: [具体修改内容]
- **验证**: [验证步骤]

## 依赖关系
- 任务2 依赖 任务1
- 任务3 依赖 任务1 和 任务2

## 预计时间
[总时间]
```

## 触发条件

此技能在以下情况自动触发：

1. 用户批准了设计方案
2. 用户说"开始实施"或"go"
3. 存在 `.trae/documents/` 目录下的设计文档

## 与其他技能的配合

- **brainstorming** - 产生设计文档
- **subagent-driven-development** - 执行计划中的任务
- **test-driven-development** - 实现时采用TDD
- **requesting-code-review** - 完成后进行代码审查
